# Шпаргалка по проекту Helpdesk

---

## Быстрый запуск

Запустить `start.bat` в корне проекта — он сам откроет два окна cmd (бэкенд + фронт) и браузер на `http://localhost:5173`.

Или вручную, два отдельных окна cmd:
```cmd
:: Окно 1 — бэкенд
cd helpdesk\backend
dotnet run

:: Окно 2 — фронтенд
cd helpdesk\frontend
npm run dev
```

## Безопасность — вопросы и ответы (8 баллов + часть из 20 за защиту)

> По критериям PDF: **аутентификация, разграничение доступа, защита данных**

---

### 1. Как реализована аутентификация? (ОБ-1)

**JWT (JSON Web Token).** При входе сервер выдаёт подписанный токен — фронт хранит его в `localStorage` и отправляет в каждом запросе.

```csharp
// AuthService.cs — GenerateToken()
var claims = new[]
{
    new Claim(JwtRegisteredClaimNames.Sub, user.Id.ToString()),
    new Claim(ClaimTypes.Name, user.Username),
    new Claim(ClaimTypes.Role, SerializeRole(user.Role)),
};
var token = new JwtSecurityToken(
    issuer: _config["Jwt:Issuer"],
    expires: DateTime.UtcNow.AddHours(24),
    signingCredentials: creds  // HMAC-SHA256 подпись
);
```

Фронт добавляет токен в каждый запрос:
```js
// client.js
headers['Authorization'] = 'Bearer ' + token
```

Бэкенд проверяет подпись автоматически через `[Authorize]` — если токен невалидный или истёк, возвращает **401 Unauthorized**.

**Почему JWT, а не сессии?**
> JWT stateless — сервер не хранит сессии, это важно для Railway где контейнер может перезапуститься. Токен самодостаточен: всё что нужно — внутри него.

---

### 2. Как реализовано разграничение доступа по ролям? (ОБ-2)

Три роли: `applicant`, `executor`, `admin`. Роль зашита в JWT-токен при выдаче и проверяется на каждом запросе атрибутом `[Authorize(Roles = "...")]`.

```csharp
// AdminController.cs — только для admin
[Authorize(Roles = "admin")]
public class AdminController ...

// TicketsController.cs — только executor и admin могут назначать
[HttpPatch("{id:int}/assignee")]
[Authorize(Roles = "executor,admin")]
public async Task<IActionResult> Assign(...)
```

Если роль не совпадает — ASP.NET Core сам вернёт **403 Forbidden**. Логику проверять не нужно — фреймворк делает это до вызова метода.

**Что если фронт подделает роль?**
> Роль берётся из токена, а токен подписан секретным ключом на сервере. Без ключа подделать подпись невозможно. Если токен изменить — подпись не совпадёт и сервер вернёт 401.

---

### 3. Как хранятся пароли? (ОБ-8)

**BCrypt** — необратимый хеш с солью. В базе никогда не хранится открытый пароль.

```csharp
// AuthService.cs — при регистрации
user.PasswordHash = BCrypt.Net.BCrypt.HashPassword(req.Password);

// при входе — сравниваем введённый пароль с хешем
bool ok = BCrypt.Net.BCrypt.Verify(req.Password, user.PasswordHash);
```

**Почему BCrypt, а не MD5/SHA?**
> MD5 и SHA — быстрые алгоритмы, их можно перебирать миллиардами в секунду. BCrypt намеренно медленный (work factor 12 — ~250мс на проверку), что делает перебор нецелесообразным. Плюс BCrypt автоматически добавляет уникальную соль к каждому паролю — одинаковые пароли дают разные хеши.

---

### 4. Защита от SQL-инъекций (ОБ-8)

**EF Core** использует параметризованные запросы — SQL-инъекция невозможна в принципе.

```csharp
// EF Core генерирует: WHERE "Email" = @p0
// @p0 передаётся отдельно, не вставляется в строку
var user = await _db.Users.FirstOrDefaultAsync(u => u.Email == req.Email);
```

Пользователь мог бы написать в поле email: `' OR '1'='1` — EF Core передаст это как параметр, а не как часть SQL. Запрос не взломается.

**Где это проверить?**
> Показать вкладку Network в DevTools: запрос идёт на `/api/auth/login` с JSON телом. Никакого сырого SQL со стороны клиента нет.

---

### 5. Защита от XSS (межсайтовый скриптинг)

React автоматически экранирует все данные перед отображением — вставить `<script>` через поле тикета невозможно.

```jsx
// React экранирует автоматически — это безопасно:
<p>{ticket.description}</p>
// даже если description = "<script>alert(1)</script>"
// в DOM попадёт текст, а не тег
```

В проекте нигде не используется `dangerouslySetInnerHTML` — все данные рендерятся через JSX.

### 6. Серверная валидация входных данных (ОБ-4)

### 9. CORS — зачем и как настроен

CORS (Cross-Origin Resource Sharing) — браузер по умолчанию блокирует запросы с одного домена на другой. У нас фронт на web.ru, бэк на Railway — разные домены.

```csharp
// Program.cs
builder.Services.AddCors(options =>
{
    options.AddPolicy("AllowFrontend", policy =>
    {
        policy.AllowAnyOrigin()   // разрешаем запросы с любого домена
              .AllowAnyMethod()   // GET, POST, PATCH, DELETE
              .AllowAnyHeader();  // Authorization, Content-Type
    });
});
```

**Не опасно ли `AllowAnyOrigin`?**
> Для учебного проекта — нормально. В продакшне лучше явно указать домен фронтенда. CORS защищает от атак через чужой сайт в браузере, но API всё равно защищено JWT-токеном — без токена ни один запрос не пройдёт.

## Возможные вопросы и ответы

**Где хранится база данных?**
> SQLite файл `helpdesk.db` в папке `backend/`. На Railway — в `/app/helpdesk.db` внутри контейнера.

**Почему SQLite а не PostgreSQL?**
> Для учебного проекта SQLite достаточен — не требует отдельного сервера. EF Core позволяет сменить провайдер одной строкой в `Program.cs`.

**Как работает авторизация?**
> При логине `AuthService.GenerateToken()` выдаёт JWT. Фронт хранит его в `localStorage` и добавляет в каждый запрос: `Authorization: Bearer <token>`. Бэкенд проверяет подпись через `[Authorize]` (ОБ-1, ОБ-8).

**Как защищены роли?**
> Атрибут `[Authorize(Roles = "admin")]` на методе — ASP.NET Core сам проверяет claim роли в токене и вернёт `403` при несоответствии (ОБ-2).

**Что такое Railway?**
> Облачная платформа. При каждом `git push` в GitHub Railway пересобирает Docker-контейнер и перезапускает бэкенд автоматически.

**Почему фронт и бэк раздельно?**
> Два независимых приложения по заданию. Фронт — статичные файлы на Apache (web.ru), бэк — REST API в Docker на Railway. Общаются через HTTPS.

**Как тесты проверяют бизнес-логику?**
> `tests/StateMachineTests.cs` — тесты таблицы переходов (допустимые и недопустимые).
> `tests/SlaServiceTests.cs` — тесты расчёта дедлайна, паузы, нарушения.
> Всего 37 тестов, запустить: `dotnet test` в папке `tests/`.
